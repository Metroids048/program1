# AI 求职台产品流程审计

日期：2026-06-15

## 结论

当前版本已经从纯前端本地规则工具升级为本地前后端分离原型：JD 工作台、实时助手、模拟面试、问题库、面试记录和简历中心可以跑通，后端提供 SQLite 存储、AI 编排、prompt registry、搜索工具抽象和 SSE 提词卡接口。

但按实时面试 Copilot 类产品的市场标准，仍只能定位为“本地可用内测版”，不能包装成完整商用版。核心差距是：浏览器语音识别仍依赖用户主动授权和 Web Speech；联网搜索需要配置 Tavily/Bing/SerpAPI key；模型能力需要 DeepSeek key；没有云账号、多端同步、桌面系统音频捕获和生产级 E2E。

## 竞品基准

- Final Round AI：强调实时 Interview Copilot、简历/JD 个性化、模拟面试和反馈。
- LockedIn AI：强调实时听取、低延迟提示和面试场景覆盖。
- Interviews.chat：强调 live transcription、real-time suggestions、面试记录。
- OfferGoose：强调 AI mock interview、动态追问和实时反馈。

本项目应对标的是“实时转写问题 -> 结合 JD/简历/题库/搜索 -> 快速生成提词卡 -> 保存记录和复盘”，不是只复制导航或做静态题库。

## 首次进入流程

1. 用户进入首页，默认看到 `JD 工作台`，不是空看板或营销页。
2. 用户粘贴 JD 后，系统生成岗位分析、匹配度、命中证据、风险缺口和自动问题库。
3. 用户可以进入 `实时助手`，通过浏览器麦克风或文字输入面试官问题。
4. 系统先用本地规则即时给出提词卡，再请求后端 SSE 生成模型/工具编排结果。
5. 用户可保存为实时助手记录，或把提词卡沉淀到问题库。
6. 用户可以进入 `模拟面试`，由 AI 面试官文字/语音出题，多轮回答后生成绑定本次 transcript 的记录和复盘。
7. `面试记录`按记录进入详情，不读取全局泛化报告。

## 模块审计

### JD 工作台

已完成：
- 首页固定为 JD 工作台。
- JD 解析会生成匹配诊断、命中证据、缺口和准备建议。
- 问题库随诊断自动生成。
- 前端会调用 `/api/positions/analyze`，后端不可用时本地规则兜底。

缺口：
- 后端诊断目前仍主要复用本地规则引擎，DeepSeek prompt registry 已存在但 JD 诊断还未完全改成结构化模型输出。
- 公司/岗位最新信息需要搜索 key 才能启用。

### 实时助手

已完成：
- 三栏工作台：问题输入、提词卡、上下文命中。
- 支持浏览器麦克风 Web Speech 和文字输入兜底。
- `/api/copilot/cue-card/stream` 使用 SSE 返回阶段进度和最终提词卡。
- 提词卡包含回答策略、开场句、要点、证据、风险提醒和追问准备，不输出完整逐字稿。

缺口：
- 不做桌面系统音频捕获，不抓会议软件系统声。
- Web Speech 在部分浏览器和网络环境不可用。
- 未实现 token 级流式模型输出，当前是阶段式 SSE。

### 模拟面试

已完成：
- 对话式多轮体验，题目来自问题库、JD/简历和临场追问。
- 可按需生成提词卡。
- 结束后保存模拟记录和绑定报告。

缺口：
- 后端 mock session API 已有，但前端仍以本地状态为主，后续可进一步让题序、追问和评分完全走后端 orchestrator。
- 语音朗读依赖浏览器 SpeechSynthesis。

### 问题库

已完成：
- 独立高频复习资产页。
- 支持诊断题、手动题、实时助手/模拟沉淀题。
- 支持编辑、重点标记、复盘备注和证据引用。

缺口：
- 暂无跨岗位全局问题聚合视图。
- 暂无 spaced repetition 复习计划。

### 面试记录

已完成：
- 支持实时助手记录和模拟面试记录。
- 保存 transcript、cueCards、speechMetrics、report。
- 记录详情绑定当次 transcript 和提词卡。
- 后端提供 `/api/records`、`/api/records/:id`。

缺口：
- SQLite 中 transcript 目前作为 record JSON 存储，尚未拆成独立 transcript turns 查询视图。
- 未做录音文件存储。

### 我的简历

已完成：
- 简历、证据库和亮点卡作为 JD/实时助手/问题库/模拟面试底座。
- 支持 PDF/DOCX/TXT/Markdown 导入。

缺口：
- 简历证据抽取 prompt 已登记，后续应让后端模型参与证据抽取并做结构化校验。

## 后端与 AI 编排审计

已完成：
- `server/`：Fastify + TypeScript。
- SQLite 表：`app_state`、`cue_cards`、`interview_records`、`search_results`、`prompt_runs`。
- AI Orchestrator：profile/JD/question bank context、cue card、follow-up、mock、record、search。
- Prompt registry：JD 诊断、提词卡、追问、模拟面试官、复盘报告、搜索摘要、简历证据抽取。
- SearchTool：Tavily/Bing/SerpAPI，可禁用并明确返回 disabled。
- LLM：DeepSeek JSON 调用，未配置或 JSON 不合法时回退本地规则。

仍是阻塞项：
- 无生产账号系统和云同步。
- 无服务端鉴权、限流、审计日志。
- 无真正桌面音频捕获。
- 无真实搜索 key 时无法联网检索。
- 无模型输出 schema 重试链路，只做一次解析和 fallback。

## 下一步优先级

1. 把 JD 诊断、简历证据抽取、复盘报告改成后端模型结构化输出，并加 schema 重试。
2. 增加 Playwright E2E：首次进入 -> 粘贴 JD -> 生成提词卡 -> 保存记录 -> 打开报告。
3. 给后端加 `/api/state` 前端冷启动同步，减少 localStorage 与 SQLite 分叉。
4. 配置真实 DeepSeek 和 SearchTool key 做低延迟测试，记录首包和完整响应时间。
5. 拆分 transcript 表并记录每轮问题来源、cue card ID、搜索引用。
6. 如要商用，另起账号/云同步/权限/加密/日志/限流方案。
