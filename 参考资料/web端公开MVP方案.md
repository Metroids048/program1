# AI 求职台公开 MVP 开发方案

日期：2026-06-18

目标：面向中国大陆市场，建设一个可推广、可注册、可稳定使用的公开 Web MVP。

## 一、版本定位

这不是完整商用版，也不是纯原型展示版，而是“可公开投放的小步快跑 MVP”。

版本目标：

- 用户可以注册登录并保存自己的求职准备数据。
- 用户可以上传简历、项目资料、JD 和补充材料。
- 系统能够基于资料解析、RAG 检索和网络搜索生成更可信的回答与模拟结果。
- 核心功能链路稳定可用，失败可降级，状态透明。

不在本版本首发范围：

- 团队协作。
- 多租户后台。
- 完整商业计费系统。
- 复杂权限角色体系。
- App 或小程序正式版本。

## 二、产品范围与页面边界

### 主流程页面

- 首页：开始准备某个岗位。
- 实时助手：接问题、生成回答框架、追问补充。
- 模拟面试：练习、追问、总结。
- JD 分析：理解岗位要求、识别准备重点。
- 题库：沉淀问题、答案、笔记、复习状态。
- 简历：维护内容、优化表达、沉淀证据。
- 记录：保存与复盘面试过程。

### 外围页面

- 注册 / 登录。
- 账号设置。
- 帮助中心。
- 隐私与服务说明。
- 反馈入口。
- 额度与试用状态。

## 三、账号与身份方案

首发推荐方案：**手机号验证码登录 + 游客体验降级入口**。

原因：

- 更符合中国大陆用户习惯。
- 比邮箱密码更轻量。
- 便于后续接试用额度、风控和运营触达。

### 用户对象拆分

- `user_account`
  - 登录身份、手机号、状态、创建时间。
- `candidate_profile`
  - 用户的候选人档案、基础信息、简历摘要。
- `job_workspace`
  - 某个岗位的准备空间。
- `knowledge_asset`
  - 上传资料、解析结果、问题、证据、记录等知识单元。
- `interview_session`
  - 实时助手或模拟面试过程。
- `interview_record`
  - 复盘结果与归档数据。

### 游客模式

允许游客在本地先试用一小段链路，但限制：

- 不提供云端持久化。
- 不保留多端同步。
- 上传资料数量受限。
- 高成本 AI 功能受限。

## 四、AI 能力架构

原则：**前端只负责交互，后端统一编排 AI。**

### 前端负责

- 文本输入与编辑确认。
- 文件上传与状态展示。
- AI 结果渲染。
- 来源状态回显。
- 失败提示与降级提示。

### 后端负责

- 简历解析。
- JD 解析。
- 项目/资料解析。
- RAG 检索与召回排序。
- 网络搜索增强。
- 提词卡生成。
- 模拟面试追问、评分、总结。
- 记录存储与导出。

### AI 返回统一结构

每次 AI 结果必须带：

- `run_type`
  - `model`
  - `rag_model`
  - `local_fallback`
- `status`
  - `success`
  - `partial_success`
  - `failed`
- `reason`
  - 失败或降级原因。
- `references`
  - 引用的内部资料与外部搜索来源。
- `latency_ms`
  - 耗时。

这样用户和前端都能知道：

- 数据从哪里来。
- 是否真的用了模型。
- 是否用了外部搜索。
- 失败时为什么降级。

## 五、资料解析与知识底座

这是本版本最关键的基础设施。

### 统一上传对象

- 简历。
- 项目文档。
- 岗位 JD。
- 自定义补充资料。

### 统一知识单元

首发固定为以下类型：

- 基本档案。
- 经历片段。
- 项目证据。
- 技能标签。
- 岗位要求。
- 问题沉淀。
- 面试记录。

### 每条知识必须保存

- `asset_id`
- `user_id`
- `job_workspace_id`
- `asset_type`
- `source_type`
  - 手填 / 上传文件 / 历史记录 / 外部搜索
- `source_file_id`
- `summary`
- `raw_excerpt`
- `tags`
- `created_at`
- `updated_at`

### 数据流

1. 用户上传文件或填写内容。
2. 后端解析生成结构化内容。
3. 结构化内容写入知识底座。
4. 同时生成切片与索引。
5. 实时助手 / 模拟面试 / JD 分析调用统一检索层。

## 六、RAG 与网络搜索方案

### RAG 方案

首发采用轻量路线：

- 结构化切片。
- 向量召回。
- 元数据过滤。
- 重排序。

不追求首发做复杂 Agent 链，而是先保证：

- 检索有边界。
- 召回有来源。
- 结果能解释。

### 检索优先级

1. 当前岗位绑定资料。
2. 用户全局资料库。
3. 历史面试记录。
4. 外部网络搜索。

### 搜索使用范围

只作为增强，不作为强依赖。首发主要用于：

- 公司背景。
- 岗位业务信息。
- 行业术语。
- 最新公开信息补充。

### 搜索结果要求

- 必须标记为“外部补充信息”。
- 必须与用户自有资料区分展示。
- 搜索超时要能降级。
- 搜索失败不能阻断主流程。

## 七、服务端与基础工程

### 服务组件

- Web API 服务。
- 解析任务服务。
- 检索与召回服务。
- 模型编排服务。
- 对象存储。
- 关系型数据库。
- 向量索引。
- 日志与监控。

### 推荐首发形态

- API：Node.js / Fastify。
- 数据库：PostgreSQL。
- 对象存储：兼容 S3 的云存储。
- 队列：轻量任务队列。
- 向量方案：Postgres 向量扩展或独立向量库二选一，优先选更轻的方案。

### 风控与稳定性

- 登录限流。
- 短信验证码频控。
- 上传大小限制。
- 文件类型白名单。
- 搜索超时降级。
- 模型调用失败重试。
- 高成本调用配额控制。

## 八、成本控制

首发必须从架构上区分不同成本层级。

### 高成本调用

- 模拟面试总结。
- 深度 JD 分析。
- 长文本资料融合回答。

### 低成本调用

- 简单提词卡润色。
- 轻量追问。
- 结构化摘要整理。

### 成本控制策略

- 试用用户限制高成本调用次数。
- 游客模式仅开放部分能力。
- 搜索与模型分层触发，不默认全量调用。
- 每次调用记录成本标签和耗时。

## 九、失败降级策略

### 登录失败

- 验证码错误、频控命中时明确提示。

### 解析失败

- 文件保留上传记录。
- 标记为待重试。
- 允许用户手动补录。

### 检索失败

- 降级到当前页面已有资料与本地结构化信息。

### 搜索失败

- 明确提示未使用外部补充信息。

### 模型失败

- 明确显示本地练习模式或基础回答模式。
- 不得伪装为模型成功。

## 十、实施顺序

### P0

- 账号与鉴权。
- 正式数据库。
- 对象存储。
- 统一知识底座模型。
- 简历/JD/资料上传与解析。

### P1

- 检索层。
- 实时助手接 RAG。
- 模拟面试接 RAG。
- 搜索增强。

### P2

- 记录复盘与资料回流。
- 试用额度。
- 监控日志。
- 反馈与帮助页面。

## 十一、验收标准

每个核心模块都必须回答四个问题：

- 数据从哪里来。
- 谁负责处理。
- 用户在哪里触发。
- 失败时怎么降级。

如果某个功能答不上这四个问题，就不能算完成。
