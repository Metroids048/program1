# AI 求职台 · UI/产品/结构全面重设计方案

基于对项目 README、AGENTS.md、vibe.md 以及整体代码结构的完整阅读，下面先给出产品诊断，再给出各功能页的完整设计方案。

---

## 一、项目诊断：优点与核心问题

**值得保留的优点：** 功能闭环是清晰的——"导入JD → 实时助手 → 生成提词卡 → 保存复盘"这条主线逻辑上完全成立；后端架构（SQLite + SSE流式提词卡 + DeepSeek）技术上是合理的；诚实状态（本地练习/模型生成）的设计理念也是对的。

**核心问题（产品角度）：**

1. **首页即仪表盘病**：AGENTS.md 里明确写了"首页第一动作是听取或输入面试官问题"，但实际实现的是一个铺满JD诊断信息的工作台——用户在最紧张的面试开始时要先到处找入口。
2. **信息密度失控**：文本堆积是因为没有建立视觉层级，所有内容都在争抢同等权重。竞品（Final Round AI、LockedIn AI）的界面在面试中只保留"问题输入区 + 提词卡区"两块，极度克制。
3. **字号体系混乱**：没有统一的字型比例（Type Scale），导致不同模块字号不协调、堆积感强。
4. **文本框与页面比例失调**：输入框没有按内容意图定义尺寸——应该"语音输入区"大、"搜索框"小，现在是反的。
5. **功能分页粒度太粗**：四个大组件（live/context/home/records）各自内部过于臃肿，每个"页面"承担了太多职责，应该拆细并让每页职责单一。

---

## 二、设计系统基础规范

先建立整个产品的设计语言，所有页面从这里推导。

**颜色** — 单色系打底 + 两个功能色
- 主背景：`#F7F7F5`（暖白）
- 卡片/面板：`#FFFFFF`
- 边框：`#E8E6E0`（0.5px）
- 主文本：`#1A1A18`
- 次文本：`#6B6B67`
- 辅助文本：`#9E9E99`
- 主色（行动/状态激活）：`#1D6F4E`（冷绿，专业感）
- 警告/录音状态：`#C4411C`（暖红）
- 信息蓝：`#2B5BA8`

**字号体系（Type Scale）**
- 页面标题（H1）：20px / 500
- 模块标题（H2）：16px / 500
- 正文：14px / 400
- 标注/标签：12px / 400
- 微标注：11px / 400

**间距节奏**：4px 基准，使用 8/12/16/24/32/48px

**关键原则**：每页只有一个"主视觉焦点"，其他元素靠色彩权重自然退到背景。

## 四、各功能页详细设计方案

### 页面一：岗位台（首页）

**设计意图：** 这是用户进来的第一页。目标不是展示信息，而是让用户在5秒内完成"选择或新建岗位"，然后快速跳入实时助手。

**结构设计：**
- 顶部导航栏：横向5个Tab，当前页高亮；右上角状态指示（后端在线/本地模式）
- 主内容区（居中，最大宽度720px）：①岗位卡列表（紧凑横向卡片，每张卡只显示公司名/岗位名/日期/状态）；②固定底部"+ 新增岗位"按钮
- 新建岗位流程：半页抽屉（Drawer）弹出，分两步——第一步粘贴JD，第二步确认岗位卡信息

**UI设计要点：**
- 岗位卡：白底卡片，左侧4px颜色条区分状态（准备中=蓝/面试中=绿/已结束=灰），一行显示"公司名 · 岗位名"，右侧显示日期和"开始助手"按钮
- 卡片不超过3行，不显示JD全文、不显示分析报告
- 选中的岗位卡片变为激活态（带绿色左边框），整个界面其他内容不变

**产品设计要点：**
- 新建岗位的JD输入框：全宽、高度自适应、最大300px，占据抽屉2/3面积——它是这一步的核心动作
- AI分析状态在抽屉内展示（加载中→岗位卡预览→确认），不跳页
- "开始面试助手"CTA按钮放在每张岗位卡最右侧，而不是在详情页

------

### 页面二：实时助手驾驶舱（核心页）

**设计意图：** 这是整个产品最重要的页面，用户在面试过程中使用。界面必须极度克制——用户处于高压状态，不能有任何认知干扰。参考 LockedIn AI 的驾驶舱布局：左侧语音区、右侧提词卡。

**结构设计（1280px宽度）：**
- 顶部状态栏（高度36px）：当前岗位名 + 面试计时器 + 录音状态指示 + 结束按钮
- 左侧主区（宽度55%）：语音/文字输入区域，占绝大多数空间
- 右侧提词卡区（宽度45%）：当前提词卡滚动展示

**UI设计要点：**
- 语音输入区：大圆形录音按钮居中，已识别文本显示在按钮上方（半透明浅灰背景，14px，行高1.8），识别中文字闪烁
- 提词卡：白底卡片，内部分区明确——策略标签（绿色小标签）、开场句（16px加粗）、要点列表（14px，前置"·"）、证据（橙色标注）、风险提醒（红色小字）、可能追问（灰色折叠）
- "手动确认"和"自动生成"两个模式切换放在顶部状态栏，不占主区空间
- AI模型状态徽章：在提词卡右上角，3种状态（模型生成=绿/本地练习=橙/生成中=灰动态）

**产品设计要点：**
- 提词卡是输出，语音是输入——两者的视觉权重要清晰区分，不能平等
- 历史提词卡向下滚动，当前卡片固定在右侧顶部
- 进度感：顶部有当前已处理题数（如"3/? 题"）---
参考：page2_live_assistant_wireframe.html
### 页面三：模拟练习间

**设计意图：** 模拟练习是"正式面试前的预演"，视觉氛围要比实时助手"练习感"更强，而不是完全照搬驾驶舱。用户心态不同——这里可以犯错、可以重来。

**结构设计：**
- 顶部：题目进度条（线性，当前题/总题数）
- 中间主区：两列——左侧"题目卡"（显示当前问题 + 意图说明），右侧"回答区"（语音/文字输入）
- 底部操作区：跳过/重录/提交回答/结束练习

**UI设计要点：**
- 题目卡：浅紫色背景，题型标签（行为类/技术类/动机类），问题大字展示（18px），下方折叠显示"考察意图"
- 回答区：浅色输入框，左下角录音按钮（比实时助手小，因为这里时间充裕）
- 提交后：右侧出现AI追问（灰色气泡），不是卡片形式——更像对话
- 进度条：细线条，当前题高亮，完成题打勾

**产品设计要点：**
- 模拟练习支持"预设题序"（来自JD和问题库），不是随机的
- "提交后生成追问"区分本地/模型，本地追问用"本地练习"标签，样式更淡
- 结束后自动跳转到"练习报告"子页---
参考页面：page3_mock_interview_wireframe.html
### 页面四：上下文资料

**设计意图：** 这是"材料库"页，用户在面试前准备或面试后查阅。内容多，但都是辅助性的，不是操作中心。用Tab在子页间切换，每个Tab只做一件事。

**结构设计：**
- 左侧固定侧边栏（宽度200px）：当前岗位信息摘要 + 子页面Tab导航
- 右侧内容区（自适应宽度）：三个子Tab —— JD解析 / 简历证据 / 问题库

**UI设计要点：**
- JD解析Tab：两栏——左侧原始JD文本（只读，等宽字体小字），右侧解析结果（岗位要求标签、关键词高亮、匹配度）
- 简历证据Tab：证据条目列表（每条证据一行：项目名 + 关键成就 + "命中要求"标签），支持手动新增
- 问题库Tab：按类型分组的题目列表（行为类/技术类/动机类），每题可以点击查看"模型参考框架"

**产品设计要点：**
- 上下文资料是所有页面的"底座"，但不应该在面试过程中被打开——它是准备阶段使用的
- 问题库的题目序列就是模拟练习的题序，两者联动
- 简历证据的条目会出现在提词卡的"证据"区块---

### 页面五：记录复盘

**设计意图：** 这是面试结束后的"反思室"。左侧是记录列表（时间倒序），右侧是当前选中记录的详情报告。界面要稳重、清晰，适合慢慢阅读。

**结构设计：**
- 左侧列表（宽度240px）：面试记录卡片（时间+岗位+时长+类型），分为"实时记录"和"模拟记录"两组
- 右侧报告区（自适应宽度）：分四个折叠区块——总体概况/题目回顾/表达指标/改进建议

**UI设计要点：**
- 记录列表：极简样式，只显示时间和岗位名，当前选中记录高亮（左边框+背景色）
- 报告总体概况：数字卡片2x2（处理题数/平均回答时长/证据命中率/追问次数）
- 题目回顾：时间轴形式，每题一行（题目摘要 + 提词卡标签 + 回答时长），点击展开该题详情
- 改进建议：AI生成的3-5条具体建议，每条有"参考做法"折叠项

**产品设计要点：**
- 报告绑定当次transcript，但transcript不直接展示（太长），提供"查看完整transcript"的折叠入口
- 实时记录和模拟记录的报告模板略有不同（实时记录侧重提词卡命中，模拟记录侧重追问质量）
- 数据导出功能放在报告右上角（导出PDF / 导出JSON）---
参考页面：page5_records_review_wireframe

第一部分：项目诊断

1.1 产品现状梳理

产品定位（已明确，方向正确）

面向应届 / 实习 / 校招用户的 AI 面试助手，核心链路：

导入 JD/简历 → 开启实时助手 → 识别问题 → 生成提词卡 → 保存记录 → 复盘改进

竞品参照（研究报告已列出）

Final Round AI、LockedIn AI、Interviews.chat、OfferGoose

技术现状（功能已跑通）


前端：React + TypeScript + Vite，CSS 占 18%
后端：Node.js + SQLite + SSE 流式提词卡
AI：DeepSeek API，本地兜底机制完整
组件结构：live.tsx / context.tsx / home.tsx / records.tsx / shared.tsx



1.2 核心问题诊断

问题一：首页即仪表盘病（严重）

AGENTS.md 明确要求"首页第一动作是听取或输入面试官问题"，但实际实现是 JD 诊断工作台。

用户在面试开始的紧张时刻需要找入口——这是最危险的 UX 失误。

根因：home.tsx 把岗位配置、JD 分析、简历诊断全部堆在首屏，没有区分"准备态"和"战斗态"两种用户状态。

问题二：信息密度失控（严重）

所有文本权重相同，没有视觉层级。提词卡、简历证据、JD 文本、状态信息争抢同等空间。

对比竞品：LockedIn AI 在面试中只保留"问题输入 + 建议输出"两块，极度克制。

根因：没有建立设计系统，每个组件自定义样式，缺乏统一的视觉语法。

问题三：字号体系混乱（中等）

没有统一 Type Scale，导致不同模块字号随意，读起来"闹"。

具体表现：提词卡正文和页面标题字号差距不足，标签字号和正文接近。

问题四：文本框/输入框比例严重失调（中等）

语音识别结果显示区（最重要的输入展示）空间过小；

JD 输入框（一次性操作）占用过多空间。

原则：空间大小 = 操作频率 × 重要程度。

问题五：组件颗粒度太粗（中等）

四个大组件各自承担太多职责，导致每个"页面"内部过于臃肿、逻辑难以维护。

live.tsx 一个文件同时处理：实时助手 + 提词卡 + 模拟面试间——这是三个独立场景。


1.3 产品优点（保留）


核心闭环逻辑正确：准备 → 实战 → 复盘
诚实状态机制（本地练习 vs 模型生成）是正确的产品价值观，应强化而非弱化
数据持久化（SQLite + localStorage 双兜底）架构合理
Web Speech API 的 interim/final/editable 分离设计正确
本地优先哲学与目标用户（学生/应届生）契合

模型生成    绿色徽章

2.7 设计原则


每页一个焦点：每个页面只有一个视觉主焦点（最大、最亮、最显眼的元素就是用户下一步要做的事）
空间 = 重要性：元素占用的面积与其操作频率和重要程度成正比
颜色编码信息：颜色不是装饰，每种颜色对应确定的含义，不混用
状态诚实：AI 模型状态（生成中/模型生成/本地练习）永远可见，不隐藏，不美化

3.3 用户流程设计

主流程（面试当天）

岗位台 → 选择岗位 → 点击"开启助手" → 实时助手页
实时助手页 → 面试官问题 → 生成提词卡 → 回答 → 重复
结束面试 → 自动保存记录 → 跳转复盘报告

准备流程（面试前一天）

提词卡区规范

宽度：45%

   - 左：框架标签徽章（11px / #2D9D6B / 主色背景）

产品逻辑


自动生成模式：停止录音后 0.5s 自动触发生成（不需要手动点击）
手动确认模式：停止录音后只显示"生成提词卡"按钮，不自动触发
生成超时（10s）：降级到本地规则，UI 显示"本地练习"状态
历史卡片向下堆叠，可以滚动查看，但当前卡片始终置顶

第五部分：关键交互设计

5.1 状态切换交互

实时助手中的模式切换（自动→手动）：
  点击状态栏中的"● 自动生成" → 变为"手动确认"
  不用弹窗，直接 toggle，状态立即反映

AI状态徽章动态：
  生成中：灰色 ● + 旋转动画（600ms/circle）
  完成：绿色 ● 模型生成（淡入动画）
  降级：橙色 ● 本地练习（淡入动画）
  过渡时间：150ms ease

5.2 语音转写状态

录音中（interim）：
  文字末尾有光标动画
  文字色：#9E9E99（灰色，表示"还没确认"）

识别完成（final）：
  文字变为 #1A1A18（深色）
  光标消失
  
停止录音后：
  文字保持，不清空
  只有用户点击"清空重录"才清空
  
可编辑状态：
  文字区变为可编辑 textarea
  右下角提示"已识别 · 可编辑"

5.3 提词卡生成动画

触发生成后：
  1. 旧卡片向下推移（transform: translateY，200ms）
  2. 新卡片从顶部淡入（opacity 0→1，300ms，延迟100ms）
  3. AI状态徽章旋转动画
  4. 完成后徽章渐变为静止态

注意：动画不应超过 300ms，面试中不需要过度动效
减弱运动：@media (prefers-reduced-motion) 取消所有动画

5.4 空状态设计

岗位台（无岗位）：
  中心显示虚线框
  图标：📄（大号，但不过分）
  主文案：还没有岗位，粘贴 JD 开始
  次文案：30秒生成岗位卡
  CTA：+ 新增岗位（链接样式，不需要按钮）

记录复盘（无记录）：
  主文案：还没有面试记录
  次文案：完成一次实时助手或模拟练习后自动保存
  CTA：→ 去模拟练习

禁止使用：
  × 空状态不放大段文字描述功能
  × 空状态不显示功能介绍
  × 空状态只有一句话+一个行动入口
