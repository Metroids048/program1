# Vibe Coding 工程化标准 — 多 Agent 约束与规范体系

> **定位**：比 AGENTS.md 和 memory 文件更高级别的全局工程规范。定义任务分级、Agent 行为约束、变更检测规则和持续改进机制。
> **适用范围**：所有使用 AI Agent（Cursor / Codex / Claude Code / Reasonix）的开发项目。
> **版本**：v1.0 — 2026-06-24

---

## 一、任务复杂度分级体系（L0–L4）

### 1.1 为什么需要分级

不同复杂度的任务需要不同的流程严谨度。用同一套流程处理"改一个文案"和"重构整个路由系统"会导致要么过度流程化（浪费 token），要么流程不足（引入 P0 错误）。

### 1.2 五级分类标准

| 级别 | 名称 | 典型任务 | 文件影响 | 时间估算 | 流程要求 |
|------|------|---------|---------|---------|---------|
| **L0** | 单点修复 | 改文案、修一个 CSS 颜色、改一个变量名 | 1 文件、<5 行 | <5 分钟 | 直接修改 + 肉眼确认 |
| **L1** | 局部修改 | 加一个按钮、修一个 bug、改一个组件 | 1-2 文件 | <30 分钟 | 修改 + 自测 + 确认 |
| **L2** | 功能增改 | 新增一个小功能、重构一个组件、改 API 调用 | 2-5 文件 | <2 小时 | R2T 任务卡 + 变更影响分析 + verify |
| **L3** | 跨模块变更 | 新增页面、修改路由、改共享类型、重构状态管理 | 5-10 文件 | 半天 | 完整 R2T + 影响分析 + 功能回归检查 + verify + 手动走查 |
| **L4** | 架构级变更 | 重构目录结构、换技术栈、改全局状态模型、跨项目同步 | 10+ 文件 | 1-3 天 | ADR + 方案评审 + 分 Phase 执行 + 跨端验证 |

### 1.3 自动分级规则

Agent 应根据以下信号**自动判断**任务级别：

```
触发 L4 的信号：
- 用户说"重构"、"重新设计"、"大改"
- 涉及 10+ 文件
- 修改 types.ts 中的核心接口
- 涉及数据库 schema 变更

触发 L3 的信号：
- 新增页面/路由
- 修改 router.ts
- 修改共享类型（非核心接口）
- 涉及 App.tsx 的路由分支修改

触发 L2 的信号：
- 新增/修改一个完整组件
- 跨 2-5 个文件
- 修改 API 调用逻辑

触发 L1 的信号：
- 修一个明确的 bug
- 改一个组件的一部分
- 添加一个小组件

触发 L0 的信号：
- 单文件单行/少量修改
- 纯文案修改
- 纯样式微调
```

### 1.4 分级对应的门禁要求

| 门禁 | L0 | L1 | L2 | L3 | L4 |
|------|:--:|:--:|:--:|:--:|:--:|
| 提问门禁（模糊需求时） | — | 建议 | 必须 | 必须 | 必须 |
| R2T 任务卡 | — | — | 必须 | 必须 | 必须 |
| 变更影响分析 | — | — | 必须 | 必须 | 必须 |
| 功能回归检查 | — | — | 建议 | 必须 | 必须 |
| verify 命令 | — | 建议 | 必须 | 必须 | 必须 |
| 手动走查 | — | — | 建议 | 必须 | 必须 |
| ADR 记录 | — | — | — | 建议 | 必须 |
| 跨端验证 | — | — | — | 建议 | 必须 |

## 二、Agent 行为约束矩阵

### 2.1 通用约束（所有级别适用）

| # | 约束 | 说明 |
|---|------|------|
| C1 | **禁止假完成** | 不得声称完成但实际未验证。不得用 TODO/placeholder 冒充实现。 |
| C2 | **禁止静默删除** | 删除任何用户可见的功能/代码必须告知用户。删除前必须对照功能清单。 |
| C3 | **禁止跨层混改** | 每轮只选一个主改动类型：产品主线 / IA / UI / AI·数据。 |
| C4 | **禁止脑补需求** | 模糊输入必须先提问澄清，不得自作主张。 |
| C5 | **禁止扩大范围** | 用户说"修 A"，不得顺手"优化 B"。最小 diff 原则。 |
| C6 | **必须 fresh verify** | 验证结果必须来自本轮刚运行的命令，不得复用缓存。 |
| C7 | **必须上下文注入** | 每次任务开始必须加载全局配置 + 项目配置 + 功能清单。 |

### 2.2 按级别的额外约束

**L0-L1 特别注意事项**：
- 即使是单行修改，如果涉及 A/B 级受保护功能的代码，也必须先检查功能清单
- "修一个 bug" 如果涉及 `useEffect` 块的修改，必须理解其业务用途后再动手

**L2-L3 特别注意事项**：
- 修改共享工具函数（`shared.tsx`、`copy.ts`）前必须全局搜索所有调用点
- 修改类型定义前必须检查所有消费者的兼容性
- 路由变更必须手动测试每条受影响的路由

**L4 特别注意事项**：
- 必须先写 ADR 并获得用户批准
- 必须分 Phase 执行，每个 Phase 独立验证
- 必须至少两个 Agent 工具交叉验证
- 必须在每个 Phase 完成后更新 PRD

## 三、变更影响自动检测规则

### 3.1 检测原理

Agent 在执行修改前，应通过以下步骤自动评估变更范围：

```
Step 1: 识别变更目标
  → 文件路径 + 变更类型（新增/修改/删除）+ 变更位置（函数/类型/组件）

Step 2: 搜索引用（grep）
  → 如果修改了导出 → grep 所有 import 该导出的文件
  → 如果修改了组件 → grep 所有使用该组件的文件
  → 如果修改了类型 → grep 所有使用该类型的文件

Step 3: 依赖展开
  → 对每个引用者递归执行 Step 2（最多 2 层）

Step 4: 分级
  → 1 文件 → L0/L1
  → 2-5 文件 → L2
  → 5-10 文件 → L3
  → 10+ 文件 → L4

Step 5: 功能对照
  → 查询 feature-regression-guard.md 中被影响的文件是否关联 A/B/C 级功能
  → 如果关联 A 级 → 风险等级至少 🟠
  → 如果关联多个 A 级 → 风险等级 🔴
```

### 3.2 具体检测命令

```bash
# 检测所有引用某导出的文件
grep -r "import.*from.*router" --include="*.tsx" --include="*.ts"

# 检测所有使用某组件的文件
grep -r "<JobsPage" --include="*.tsx"

# 检测所有使用某类型的文件
grep -r "AppState\|Position\|InterviewRecord" --include="*.tsx" --include="*.ts" -l
```

### 3.3 受保护代码块指纹

以下代码块具有特殊保护标记，Agent 修改前必须确认：

| 代码块 | 文件 | 关键指纹 | 保护级别 |
|-------|------|---------|---------|
| auto-cue-insert useEffect | `live.tsx` | `streamCueCardFromServer` + `normalizeCard(generateCueCard(` | A5 |
| buildInterviewReport | `interviewEngine.ts` | `completeness.*relevance.*evidenceStrength` | A6 |
| parseRoute 函数 | `router.ts` | `export function parseRoute` | A2 |
| createOrUpdatePosition | `App.tsx` | `upsertPositionIntakeOnServer` + `makeId("conv")` | A1 |
| PRIMARY_NAV 常量 | `appShell.tsx` | `{ id: "jobs", label: "岗位"` | A4 |
| useAuth hook | `auth.ts` | `export function useAuth` | A3 |

## 四、Agent 任务卡标准（跨工具统一）

### 4.1 任务卡字段规范

```markdown
# 任务卡 — TASK-{编号}

## 元信息
- **复杂度**：L{0-4}
- **主改动类型**：产品 / IA / UI / AI·数据
- **版本目标**：原型 / 内测 / MVP / 商用
- **执行 Agent**：Cursor / Codex / Claude Code / Reasonix

## 需求描述
[一句话描述 + WHY]

## 变更范围
- **修改文件**：{文件列表}
- **新增文件**：{文件列表}
- **删除代码**：{代码块 + 原因}
- **预计行数**：±{N} 行
- **预计时间**：{N} 分钟/小时

## 影响分析
- **依赖消费者**：{受影响的文件列表}
- **受保护功能影响**：{A1-C8 中受影响的编号}
- **风险等级**：🟢/🟡/🟠/🔴

## 不动清单（Out of Scope）
- [ ] 不改 {X}
- [ ] 不碰 {Y}

## 验收卡
- **必须有**：{可验证的条件}
- **禁止有**：{不能出现的退化}
- **验证命令**：{具体命令}
- **走查路径**：{手动测试步骤}

## 完成记录（事后填写）
- **实际修改**：{实际文件 + 行数}
- **验证结果**：PASS / FAIL
- **遗留问题**：{待后续处理}
```

### 4.2 任务卡生命周期

```
创建（Agent 或用户）
  → 确认（用户审批 L2+ 任务）
  → 执行（Agent 逐步实施）
  → 验证（运行 verify + 手动走查）
  → 记录（回写完成记录）
  → 归档（写入 task-history）
```

## 五、持续改进机制

### 5.1 每次返工的复盘流程

1. **分类**：需求未锁定 / 层级混改 / 验证缺失 / 配置未加载 / 上下文漂移 / 功能误删 / 跨端不一致
2. **记录**：写入 `ai-engineering-pitfalls.md`
3. **修复**：更新对应规则文件
4. **预防**：添加自动化检查（如 grep 指纹检测）

### 5.2 规则演进

- 同类问题出现 **2 次** → 在 coding-conventions 中添加检查项
- 同类问题出现 **3 次** → 提升为 global-agent-master 永久规则
- P0 问题出现 **1 次** → 立即创建受保护功能条目 + 删除门禁

### 5.3 配置健康度检查

每周建议执行一次：

```powershell
# 检查全局配置一致性
cat ~/.ai-workspace/memory/global-agent-master.md | measure-object -line
cat ~/.ai-workspace/memory/coding-conventions.md | measure-object -line
cat ~/.ai-workspace/memory/ai-engineering-pitfalls.md | measure-object -line

# 检查功能清单完整性
cat ~/.ai-workspace/memory/feature-regression-guard.md | grep "A\d\|B\d\|C\d" | measure-object -line

# 确认所有受保护功能仍可通过验证
npm run verify
```

## 六、附录：业界参考实践

### 6.1 Aider 的 Map-Reduce 模式

Aider 使用 map-reduce 架构处理大型代码库：
- **Map 阶段**：将代码库分成 chunks，每个 chunk 独立分析
- **Reduce 阶段**：汇总分析结果，生成统一修改方案

**借鉴点**：L3/L4 任务可将大型变更拆成独立的子任务（每个子任务 ≤L2），分别在不同 Agent 会话中执行，最后汇总验证。

### 6.2 Claude Code 的 Memory 体系

Claude Code 使用分层记忆：
- 项目记忆（`.github/agent/memory/`）
- 全局记忆（`~/.claude/`）
- 会话记忆（会话内上下文）

**借鉴点**：我们的三层配置体系（全局 → 项目 → 任务）正是参照此模式。

### 6.3 Cursor Rules 的文件匹配

Cursor 的 `.cursorrules` 支持 glob 模式匹配，只对特定文件类型应用特定规则。

**借鉴点**：可以为不同文件类型设定专门的 Agent 行为规则（如 `.tsx` 文件强制功能清单检查、`.ps1` 文件强制 dry-run 预览）。

### 6.4 关键原则总结

> **Vibe Coding 的本质不是"随便说"，而是"结构化协作"。**
>
> 1. **分级处理**：不同复杂度用不同流程，不过度也不欠缺
> 2. **功能清单化**：已完成的每个功能都要"注册"，防止误删
> 3. **变更可视化**：每次修改前先评估影响范围，不让修改"悄悄"破坏
> 4. **门禁自动化**：能自动检查的就不要让 Agent 手动判断
> 5. **持续改进**：每次踩坑都是完善规则的机会，不要只修 bug 不修规则
