从想法到产品：Vibe Coding 全流程工程化指南

Codex · Cursor · Claude Code 实战手册（含失控防控版）

执行摘要： 本指南在原有全流程方案基础上，系统整合了 vibe coding 三大核心失控机制（需求漂移、状态隐式化、连续生成无验收点）的解决方案。核心思路是：引入"三份强制文档"冻结系统边界，将"聊天式 coding"改造为"工程分段交付"，并在每个阶段设置可量化的验收闸门。原有的里程碑、分工、对话模板、工具分工等内容在此基础上全面升级，形成一套真正可控的 AI 辅助开发工作流。


一、为什么你会反复返工：三大失控机制

在进入流程之前，必须先理解 vibe coding 的失控根因，否则再好的模板也只是另一轮混乱的起点。

1.1 需求漂移（你的需求一直在流动）

现有模式：

想法 → prompt → 出代码 → 发现不对 → 改 prompt → 再出代码 → 再改

根本问题：


AI 每次都会"重新解释需求"，局部描述改了，系统定义没有被冻结
你改的是措辞，AI 理解的是一个新的系统


症状： 改了 A，B 坏了；看起来都对，但整体不一致

1.2 状态隐式化（系统真实结构只活在 AI 上下文里）

vibe coding 最大的坑不是代码本身，而是：


系统真实结构只存在于 AI 的聊天上下文里，而不是你可控的文档里。



经典后果：


哪个模块负责什么？你说不清
AI 自己"顺手加的"逻辑？你不知道
state / API / UI 之间的关系全是隐式的


准确总结：你不是在 debug 代码，你是在 debug AI 的假设。

1.3 连续生成无验收点（你在做流水，AI 在自由发挥）

现有节奏： 一直让 AI"继续写" → 没有任何 checkpoint → UI + API + 数据结构一起漂移 → 没有办法 rollback

AI 的默认行为：


自动补它认为合理的功能
自动加 dependency
自动"帮你优化体验"
自动加它觉得"应该有"的逻辑


它不知道你的边界在哪里，于是系统慢慢变成"看起来完整，但不断长歪的产品"。


二、核心解法：三份强制文档 + 工程分段 + 验收闸门

2.1 三份强制文档（项目启动前必须生成）

这不是可选项。 任何 vibe coding 项目，在写第一行代码前，必须先生成以下三份文档。

文档 A：冻结版 PRD（需求锁死）

不是普通 PRD，是不可随意修改的版本。

必须包含说明用户目标用一句话描述用户要完成什么核心流程3~5 个步骤，具体到页面不做什么列出明确排除的功能（这是最重要的部分）成功指标如何判断这个功能做完了

示例提示词：

请根据以下想法生成冻结版 PRD。注意：
1. 核心功能只保留 MVP 必须的，其余全部放入"不做什么"清单
2. 不做什么清单要尽量详细，防止 AI 自行扩展
3. 最多 3 个问题来澄清不明确的地方，其他直接做决定

文档 B：系统设计图（结构显式化）

哪怕是最简单的项目，也必须写这份文档。作用是防止 AI 顺手发明新系统。

# 系统设计图（示例：笔记 App）

## 前端模块
├── LoginPage        # 登录表单
├── NoteListPage     # 笔记列表（只读）
└── NoteEditorPage   # 编辑器（含保存）

## 后端 Service
├── auth-service     # 登录验证
├── note-service     # CRUD 接口

## 数据层
├── users 表         # id, email, password_hash
└── notes 表         # id, user_id, title, content, updated_at

## API 接口（固定，不允许新增）
POST /api/auth/login
GET  /api/notes
POST /api/notes
PUT  /api/notes/:id

文档 C：约束清单（最关键但最常被忽视）

在每次 prompt 前，必须将约束清单作为上下文附入。

# 本项目约束清单（AI 必须遵守）

❌ 不允许新增系统设计图之外的模块
❌ 不允许自动修改数据库结构
❌ 不允许改变 API 返回字段结构
❌ 不允许在 UI 层引入新的状态管理方案
❌ 如发现需求与现有约束冲突，必须先暂停并报告，不得自行决定

✅ 所有新增字段必须在文档 B 中先更新，再开始实现
✅ UI 状态统一在 [指定位置] 管理


三、项目里程碑与 MVP 定义

3.1 里程碑划分原则

里程碑的核心原则是：每个阶段必须可以独立验收，未通过验收不得进入下一阶段。

里程碑示意图：

[冻结文档] → [架构设计] → [UI骨架] → [API实现] → [联调粘合] → [质量收尾] → [上线]
     ↓             ↓           ↓           ↓            ↓             ↓
  三份文档      设计图确认   页面可跑    接口通过     端到端可用    测试通过

3.2 里程碑详解

里程碑核心交付物验收标准0. 冻结文档PRD + 系统设计图 + 约束清单三份文档完整，不做什么清单不为空1. 架构设计模块划分、数据结构、API 接口文档AI 只输出文档，不写任何代码2. UI 骨架所有页面的静态布局代码无 API 调用，无 state 逻辑，只有 UI3. API 实现后端接口代码不允许修改 UI，不允许新增功能4. 联调粘合glue code，前后端对接端到端核心流程跑通5. 质量收尾测试脚本、UI 修复、样式统一自动化测试通过，无视觉错位6. 上线部署CI/CD 配置、商业化方案自动构建通过，生产环境稳定

3.3 MVP 定义示例（以社交笔记 App 为例）

✅ MVP 包含：


用户登录/登出
笔记创建、查看、编辑、删除
笔记列表展示


❌ MVP 明确排除（写入约束清单）：


分享功能
推送提醒
富文本编辑器
搜索功能
多设备同步



四、工程分段交付：把 AI 从"自由发挥"变成"分工执行"

核心原则：一次 prompt 只做一件事。

4.1 四步分段法

Step 1：只做设计（禁止写代码）
"先不要写代码，只输出模块划分 + 数据结构 + API 接口列表"

Step 2：只做 UI（锁定：无 API、无 state 逻辑）
"只实现页面静态布局，不要调用任何接口，不要添加状态管理"

Step 3：只做 API（锁定：不改 UI、不加功能）
"只实现以下接口，不允许修改前端代码，不允许新增接口"

Step 4：做 Glue Code（联调粘合）
"将以下 UI 与以下 API 对接，不允许修改接口结构"

4.2 每步必须附带的约束语句

在任何 prompt 前，都需要先附上文档 B（系统设计图）和文档 C（约束清单），然后再描述任务。

固定 prompt 结构：

[粘贴约束清单]
[粘贴当前系统设计图]

当前阶段：Step X - [阶段名称]
本轮任务：[具体且单一的任务描述]
本轮禁止：[明确不允许做的事]
完成标志：[如何判断这轮任务完成了]


五、验收闸门：每个阶段的通过条件

这是最常被跳过、也是最重要的环节。

5.1 各阶段验收检查表

阶段 0 - 冻结文档：


 PRD 中"不做什么"清单不为空
 系统设计图包含前端模块、后端 service、数据表、API 列表
 约束清单中有明确的"不允许"条目


阶段 1 - 架构设计：


 AI 只输出了文档，没有写任何代码
 所有数据表字段已确定（后续不允许修改）
 API 返回结构已固定


阶段 2 - UI 骨架：


 页面能在浏览器中打开（可以是静态 Mock 数据）
 没有任何真实 API 调用
 没有引入系统设计图之外的新模块


阶段 3 - API 实现：


 接口返回结构与文档 B 完全一致
 没有新增未记录的字段
 前端代码未被修改


阶段 4 - 联调粘合：


 端到端核心流程可以完整跑通
 没有修改 API 接口结构
 没有新增功能


阶段 5 - 质量收尾：


 关键路径自动化测试全部通过
 页面截图无明显视觉错位
 代码通过 Lint 检查



规则：任何一个 [ ] 未打勾，不进入下一步。




六、各阶段详细对话流程模板

阶段 0：冻结文档生成

输入： 你的初始想法

示例提示：

我要做一个 [X]。

请按以下格式输出冻结版 PRD：
1. 目标用户（一句话）
2. 核心痛点（一句话）
3. MVP 核心流程（3~5步）
4. 不做什么（至少 5 条）
5. 成功指标（可量化）

如信息不足，最多提 3 个问题，其他你来决定。不要添加任何我没有提到的功能。

验收： PRD 完整，"不做什么"至少 5 条


阶段 1：架构设计（只出文档，不写代码）

输入： 冻结版 PRD

示例提示：

[粘贴约束清单]

根据以上 PRD，只输出以下内容（不要写任何代码）：
1. 前端模块列表（页面 + 组件）
2. 后端 Service 列表
3. 数据表结构（字段级别）
4. API 接口列表（路径 + 请求/返回格式）

约束：不得超出 PRD 范围，发现需要新增功能时必须报告而不是自行添加。

验收： 数据结构和 API 格式已固定，后续不允许修改


阶段 2：UI 骨架

输入： 系统设计图（前端模块部分）

示例提示：

[粘贴约束清单]
[粘贴前端模块设计]

当前阶段：只实现 UI 静态骨架

请为 [页面名称] 生成前端代码：
- 使用静态 Mock 数据（不要调用任何 API）
- 不要引入状态管理库（除了已在设计图中指定的）
- 不要添加设计图中未列出的组件

完成后回答：是否有任何地方超出了系统设计图的范围？

验收： 页面可以在浏览器打开，无 API 调用


阶段 3：API 实现

输入： 系统设计图（后端 Service + 数据表 + API 接口部分）

示例提示：

[粘贴约束清单]
[粘贴 API 接口文档]

当前阶段：只实现后端接口

请实现以下接口：[列出接口]
- 严格按照接口文档中的返回结构，不得新增字段
- 不要修改任何前端代码
- 如果发现接口设计有问题，暂停并报告，不要自行修改

完成后提供：每个接口的 curl 测试命令

验收： curl 测试全部通过，返回结构与文档一致


阶段 4：联调粘合

输入： 已完成的 UI 代码 + 已完成的 API

示例提示：

[粘贴约束清单]

当前阶段：联调粘合（glue code）

将以下 UI 与以下 API 对接：
- 不允许修改 API 接口结构
- 不允许修改 UI 组件结构
- 只添加 API 调用代码和数据绑定

完成后回答：是否有任何地方修改了 UI 或 API 的既有结构？

验收： 核心流程端到端跑通


阶段 5：质量收尾

UI 优化提示：

根据以下设计规范审查当前 UI：
- 主色：[颜色值]
- 字体大小：[规格]
- 间距规则：[规格]

输出：哪些地方不符合规范（列表），然后逐一修复。不要修改任何业务逻辑。

测试生成提示：

为以下核心流程生成端到端测试脚本（使用 Playwright）：
1. [流程1]
2. [流程2]

要求：覆盖成功路径和主要失败路径。

验收： 测试全部通过，截图对比无错位


阶段 6：上线部署

示例提示：

为本项目创建 CI/CD 配置（GitHub Actions），要求：
- 每次 PR 自动运行测试
- 测试通过后自动部署到 [平台]

另外，给出两种商业化方案（订阅制 vs 广告）的简要对比，包括实现复杂度和预期收益。


七、工具分工建议（Codex / Cursor / Claude Code）

工具最适合的阶段核心优势注意事项Cursor阶段 2、5（UI 骨架、UI 优化）IDE 内联编辑、Tab 补全、实时反馈大型跨文件重构可能出错Claude Code阶段 1、3、4（架构、API、联调）200K 上下文、直接读写文件、执行命令实时交互体验不如 CursorCodex阶段 5、6（批量修改、CI/CD）并行批量执行、自动创建 PR需要足够上下文，不适合实时对话

分工黄金法则：


需要实时交互和局部编辑 → Cursor
需要理解整体项目和大范围修改 → Claude Code
需要并行批量跑任务 → Codex


上下文管理技巧：


Cursor：在 .cursorrules 中固化约束清单和设计规范
Claude Code：在 claude.md 中写入系统设计图和约束清单，每次会话自动加载
Codex：在 .codex/config.toml 中设置项目根目录信息



八、Token 与时间成本优化

8.1 高频 Token 浪费原因及对策

浪费原因解决方案节省估算每次都重新描述需求使用固定前缀（约束清单 + 设计图）30~40%一次 prompt 覆盖太多功能工程分段，每次只做一件事20~30%频繁开新会话保持会话连续，利用 Prompt Cache20%全量代码拉入上下文语义检索 MCP 插件（如 claude-context）60~75%

使用语义检索插件后，总 Token 消耗可降低约 60%。
