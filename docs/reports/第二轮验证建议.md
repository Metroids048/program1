# 第二轮验证建议

## 1. AI 主链路 success 专项

目标：

- 验证 `cue-card` / `mock answer` / `resume/ai` 在真实 DeepSeek 模型下返回 `success`
- 固化临时环境变量注入方式，不要求把密钥写回 `.env`

建议动作：

1. 用临时 `DEEPSEEK_API_KEY` 执行 `npm run test:ai-success-smoke`
2. 默认模型使用 `deepseek-v4-flash`
3. 对 `cueCard`、`mockAnswer`、`resumeAi` 的 `backendStatus = success` 做断言
4. 输出模型名、状态与耗时，作为回归证据

## 2. 独立 Search Provider 专项

目标：

- 验证 `/api/search` 在真实独立 provider 下的成功返回
- 保持与 DeepSeek 主链路 success 验收分离

建议动作：

1. 配置 `SEARCH_PROVIDER`
2. 配置对应 `SEARCH_API_KEY`
3. 增加一条正式 smoke script
4. 对 `provider != disabled`、结果条数、文案展示做断言

说明：

- 当前 `/api/search` 仍只支持 `tavily | bing | serpapi`
- DeepSeek 联网辅助不计为 `/api/search` 独立路由成功态

## 3. 浏览器渲染层专项

目标：

- 补齐 Codex 无法执行的 UI 渲染验收

建议在 Cursor / 人工浏览器完成：

1. 首页 1280x720 首屏
2. 模拟面试页 1280x720
3. 实时助手完整语音交互
4. 简历三栏布局
5. 记录页时间轴与 Transcript
6. 390px 移动端
7. 浏览器控制台 error 检查

## 4. 性能与包体专项

目标：

- 降低导入相关依赖导致的大 chunk 风险

建议动作：

1. 评估 `pdfjs-dist` 懒加载
2. 评估 `mammoth` 懒加载
3. 评估简历导入能力是否可拆到独立 chunk
4. 建立 chunk 体积阈值

## 5. 真实模型稳定性专项

目标：

- 把当前临时验证固化为可重复执行的成功态回归

建议动作：

1. 使用现有 `test:ai-success-smoke`
2. 固定验证三条链：
   - `cue-card`
   - `mock answer`
   - `resume/ai`
3. 输出：
   - `backendStatus`
   - `fallbackReason`
   - 响应长度
   - 耗时
4. 连续跑 3 次，确认不是偶发成功

## 6. 风险控制专项

目标：

- 防止产品在“真实性”上出现错误引导

建议动作：

1. 为提词卡增加输出审查
2. 对“虚构 / 编造 / 假设经历”类措辞加测试
3. 将 fallback 证据与真实证据分层展示

## 7. 测试资产建设建议

建议新增：

1. `npm run test:full-flow`
2. `npm run test:ai-success-smoke`
3. `npm run test:search-smoke`
4. 账户导入失败用例
5. 真实性 guardrail 用例
