# 公开 MVP 功能开发设计方案

更新时间：2026-06-19

## 1. 文档目的

本文件说明公开 MVP 下一阶段的核心开发设计，不讨论云端化、商业化后台或多租户，只聚焦当前仓库最需要跑通的能力闭环。

## 2. 开发主线

后续开发统一围绕以下主线推进：

1. 用户处理真实 JD
2. 系统沉淀岗位资料底座
3. 模拟面试预演
4. 实时助手生成提词卡
5. 保存记录与报告
6. 回到问题库和简历继续复盘

## 3. 功能模块设计

### 3.1 实时助手提词卡

目标：

- 把当前 `/live` 从“本地规则为主”升级成“本地即时反馈 + 后端模型覆盖”
- 保持现有页面结构和本地 fallback 体验

后端链路固定为：

1. 接收问题文本
2. 召回当前岗位上下文
3. 判断是否触发搜索
4. 调用 DeepSeek 输出结构化提词卡
5. 返回 `AnswerCueCard + AiRunMeta + evidenceTrace`

前端行为固定为：

1. 本地规则先给占位卡
2. SSE 接收进度和最终卡
3. 成功则覆盖占位卡
4. 失败则停在本地练习模式

验收重点：

- 普通问题不联网也能给出带证据的卡
- 行业/公司时效题可附搜索来源
- 不输出完整逐字稿

### 3.2 模拟面试

目标：

- 保持 `/mock` 当前交互结构
- 把出题、追问、报告统一纳入后端模型编排

设计要求：

- 首题必须结合当前岗位、简历证据和问题库
- 追问必须绑定刚刚的回答内容
- 下一题仍要围绕岗位准备重点
- 结束后报告保留 `model/local` 来源标记

与实时助手的共用能力：

- 同一个 Prompt registry
- 同一个 DeepSeek client
- 同一套 RAG 检索
- 同一套搜索策略

### 3.3 简历 AI 对话

目标：

- 把 `/resume` 右侧聊天从本地模拟建议升级为真实后端 AI 对话

支持动作固定为：

- 优化当前区块
- 优化整份简历
- 按当前岗位做匹配分析

返回内容固定为：

- 回复文案
- 可应用建议
- 来源说明
- 应用目标

前端规则：

- 左侧和中间的编辑结构不变
- 用户手动点击后才应用建议
- 保存仍以后端接口为准

### 3.4 JD / 岗位分析

目标：

- 保留现有 `/jd` 页面
- 让分析真正基于岗位 intake、简历、资料、问题和记录，而不是只靠本地组合逻辑

最低要求：

- 能解释“为什么你会收到这个面试”
- 能指出岗位重点和资料缺口
- 能输出下一步准备建议

### 3.5 RAG 资料底座

目标：

- 让问题库、简历、JD、记录真正变成模型可用的上下文资产

实现原则：

- 所有资料先归档成文档
- 再切块入 FTS 检索
- 问题时按岗位优先召回
- 结果以 evidenceTrace 形式回显

优先级顺序固定为：

1. 当前岗位项目资料
2. 当前岗位用户保存问题
3. 当前岗位上传资料
4. 简历证据
5. 历史记录

### 3.6 联网搜索

目标：

- 把搜索变成“少量但有价值”的工具，而不是所有问题都联网

触发策略固定为：

- 问题中出现明显时效性词
- 出现公司动态、行业趋势、技术版本变化

降级策略固定为：

- 搜索没配置：直接跳过
- 搜索超时：不阻塞回答
- 搜索失败：前端显示未接通，不伪装为模型已知

### 3.7 语音输入与朗读

目标：

- 维持当前 Web Speech 三态输入优势
- 补齐手动 TTS 朗读能力

设计要求：

- 输入继续只用麦克风或文字
- 停止听取后文字保留
- TTS 默认关闭
- 朗读能力只服务练习和辅助，不作为系统自动代答机制

### 3.8 记录与复盘

目标：

- 让记录既可回看，又可进入后续检索上下文

记录保存内容固定为：

- transcript
- cueCards
- speechMetrics
- report
- aiMeta

不保存：

- 原始音频

## 4. 开发顺序建议

### P0

- SQLite schema 扩展：`documents / document_chunks / retrieval_runs`
- 文档切块与增量刷新
- RAG 召回与重排
- 实时助手后端模型链路接通

### P1

- 模拟面试接入统一 RAG + 模型编排
- 简历 AI 对话后端接口
- 条件搜索接入
- TTS 手动朗读

### P2

- JD 分析升级为真实后端分析
- 运行日志细化
- 内部恢复索引工具

## 5. 工程约束

- 不新增无关依赖
- 不引入 ChromaDB/Ollama 作为 MVP 强依赖
- 不改成多用户体系
- 不改页面数量
- 不把 fallback 伪装成模型成功

## 6. 测试与验收

### 6.1 RAG

- 更新 JD、简历、资料、问题、记录后，检索文档可增量刷新
- 同岗位提问时，优先命中当前岗位资料

### 6.2 实时助手

- 本地占位卡先出现
- 模型成功后覆盖卡片
- 搜索题可带来源
- 模型失败时停留本地练习模式

### 6.3 模拟面试

- 首题、追问、下一题带岗位和证据上下文
- 报告保留来源标记

### 6.4 简历 AI 对话

- 三类动作都走后端
- 建议可应用到当前区块
- 建议带依据

### 6.5 语音与记录

- 停止听取不清空文本
- 保存后仅落结构化记录，不生成音频依赖

### 6.6 工程验证

- `npm run verify`
- 补后端单测：
  - 切块
  - 重排
  - fallback
  - 搜索超时
  - SSE 事件顺序
