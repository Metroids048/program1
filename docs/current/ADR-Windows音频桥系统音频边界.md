# ADR：Windows 音频桥系统音频边界

日期：2026-07-09

## 状态

已采纳。

## 背景

旧公开 MVP 文档把语音输入限定为浏览器麦克风和手动文本，明确不做系统音频捕获。后续 deep-research-report 方案把面试实时辅助的目标扩展为：在用户主动授权下接收腾讯会议、飞书等会议软件播放的面试官声音，并接入已有 AI + RAG 提词卡链路。

如果继续保留旧边界，产品文案、法务说明、音频桥实现和测试验收会互相冲突。

## 决策

系统音频能力只通过 Windows 音频桥提供，并作为独立顶层页面 `/audio-bridge` 暴露给用户。用户必须先登录、生成配对码、启动本地音频桥并完成授权连接，系统才允许采集会议软件播放的系统音频。

实时助手保留“系统音频”开关，但职责是消费已连接音频桥的转写结果、识别问题、生成提词卡并保存记录；连接、设备管理和故障诊断放在“会议监听”页。

## 约束

- 账号严格隔离：配对码、设备令牌、设备列表和事件订阅都绑定当前账号。
- 不自动合并访客数据：登录、退出、切换账号后清理本地身份态并重新拉取当前账号服务端快照。
- 原始音频不落盘：音频桥只向服务端 ASR 转发实时 PCM 分片，持久化内容仅限 transcript、题词卡、报告和必要诊断。
- 失败不伪装：ASR 未配置或连接失败必须展示 `ASR_NOT_CONFIGURED`、本地练习或语音服务未配置状态。
- 禁止隐蔽监听：未连接音频桥、未授权、设备被撤销或账号切换后，不得继续采集系统音频。

## 影响

README、帮助、隐私、用户可见页面和验收脚本都必须按此边界描述系统音频能力。旧文档里的“不捕获系统音频”统一替换为“仅在用户主动连接 Windows 音频桥并授权时采集系统音频”。
