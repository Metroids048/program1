# 公开 MVP 总体技术架构方案

更新时间：2026-06-19

## 1. 文档目的

本方案用于约束 AI 求职台公开 MVP 阶段的真实技术边界，避免后续开发再次走向“页面先行、能力虚空、状态混乱”的旧方向。

本方案直接映射当前仓库，不另起一套云端 SaaS 架构。

## 2. 版本边界

- 版本目标：公开 MVP
- 形态：本地优先、单机单用户
- 页面结构：保留当前多页面 IA，不合并页面
- 实时定位：练习优先，同时保留显式实时辅助
- 音频边界：默认仅麦克风和手动输入；系统音频只在用户主动连接 Windows 音频桥并授权后采集
- 数据留存：不保存原始音频，只保存 transcript、指标、提词卡、报告

## 3. 总体架构

当前公开 MVP 采用四层结构：

1. 前端 SPA
   - 技术栈：React 19 + TypeScript + Vite 7
   - 责任：页面展示、临时输入态、SSE 消费、语音输入/朗读控制
2. 本地 API 服务
   - 技术栈：Fastify 5
   - 责任：路由、模型编排、RAG 检索、搜索工具、持久化回填
3. 本地持久化与检索底座
   - 技术栈：better-sqlite3 + SQLite
   - 责任：业务快照、记录、运行日志、RAG 文档和 FTS 检索
4. 外部能力
   - OpenRouter / GitHub Models / DeepSeek：统一 AI client 内部降级
   - 讯飞 RTASR：可选流式 ASR 通道
   - Search Provider：按需启用的条件工具

## 4. 模块分层

后端统一拆成 5 层，路由名保留兼容，但实现职责明确下沉：

- `routes`
  - 对外暴露现有 API 名称
  - 只做 request 校验、响应封装、SSE 事件输出
- `services/orchestrator`
  - 负责串联 RAG、搜索、Prompt、模型、fallback
  - 不直接写 UI 文案
- `rag`
  - 负责文档切块、FTS 入库、召回、重排、evidenceTrace
- `ai`
  - 负责统一 AI client、provider 降级、Prompt registry、结构化输出修复、运行日志
- `search`
  - 负责 provider 适配、超时控制、结果归一化

前端继续保留现有页面组件，但新增能力层，避免 AI 和语音逻辑继续散落在组件里：

- `useCueCardStream`
- `useResumeAiChat`
- `useSpeechInput`
- `useSpeechOutput`

## 5. 主链路

### 5.1 首页 intake

1. 用户粘贴真实 JD 或补充岗位信息
2. 服务端更新岗位 intake
3. 同步重建岗位相关检索文档
4. 前端以服务端快照回填审核卡

### 5.2 实时助手

1. 用户通过麦克风或文本输入问题
2. 前端立即生成本地占位卡，降低等待感
3. 后端执行：
   - RAG 召回当前岗位上下文
   - 条件判断是否触发搜索
   - 调用统一 AI client 生成结构化提词卡
   - 通过 SSE 回传进度和最终卡片
4. 前端用模型卡覆盖本地卡；失败则停留在本地练习模式

### 5.3 模拟面试

1. 后端基于岗位、简历、资料和题库生成首题
2. 用户回答后，后端基于最新回答和已有上下文生成追问或下一题
3. 面试结束后生成报告并保存记录
4. 记录自动进入后续 RAG 上下文

### 5.4 简历 AI 对话

1. 用户选择当前区块、整份简历或岗位匹配分析
2. 后端拉取简历文本、证据库、当前岗位、岗位资料
3. 生成结构化回复和建议文本
4. 前端决定是否应用建议到当前编辑区

## 6. RAG 架构原则

本轮不引入 ChromaDB、Ollama、Milvus 等额外强依赖。

公开 MVP 的 RAG v1 采用 SQLite FTS5 本地检索底座，原因如下：

- 当前仓库已是单机单用户架构
- 避免要求用户额外安装独立服务
- 与现有 SQLite 快照和运行日志共用部署方式
- 对公开 MVP 的资料规模足够

RAG 输入来源固定为：

1. 当前岗位项目资料
2. 当前岗位用户保存问题
3. 当前岗位上传资料
4. 全局简历证据
5. 历史记录 transcript / report 摘要

## 7. 搜索策略

联网搜索是条件工具，不是默认主链路。

仅在下列场景触发：

- 公司最新动态
- 行业趋势
- 技术版本变化
- 明显时效性问题

搜索失败或未配置时：

- 不阻塞主流程
- 明确显示“搜索未接通”或“本地资料回答”
- 不伪装成模型已拿到外部事实

## 8. 语音与实时边界

- 输入：优先服务端讯飞 RTASR 流式转写；未配置、连接失败或浏览器录音不可用时，回退到 Web Speech API 三态转写与手动文本输入
- VAD：使用前端 Silero VAD 判断说话结束，只触发自动模式下的停止/生成，不替代手动按钮
- 输出：浏览器原生 TTS，默认关闭，仅手动触发
- 系统音频捕获只允许走用户主动配对的 Windows 音频桥，不允许后台偷听或隐蔽抓取
- 不做后台偷听式能力
- 停止听取不清空已识别文本

目标体验：

- 前端即时占位 `< 300ms`
- 本地 fallback 卡片 `< 800ms`
- 模型完整卡目标 `< 3s`

## 9. 存储与状态 owner

业务真源继续以后端为主：

- 岗位 intake
- 简历与证据
- 当前岗位资料
- 当前岗位问题
- 面试记录
- AI 运行日志
- RAG 文档与切块

前端本地只保留：

- `serverSnapshotCache`
- `uiPrefs`
- `drafts`

`app_state` 快照继续保留，用于当前前端回填兼容；RAG 和 AI runtime 数据独立存表，不塞回前端业务主状态。

## 10. 当前仓库映射

现有仓库中，以下文件和目录是后续实现的直接落点：

- `server/index.ts`
  - 保留兼容路由名
- `server/orchestrator.ts`
  - 后续应拆到服务层，避免继续膨胀
- `server/prompts/registry.ts`
  - 继续作为 Prompt 契约中心
- `server/db.ts`
  - 扩展 SQLite 查询和 RAG 数据访问
- `src/components/live.tsx`
  - 后续应迁出 AI/语音编排细节
- `src/components/resume.tsx`
  - 后续接入真正后端 AI 对话

## 11. 非目标

本轮明确不做：

- 云端多租户
- 登录注册体系
- 付费计费体系
- 未授权或隐蔽式系统音频抓取
- 原始音频持久化
- 依赖外部向量数据库的重型架构

## 12. Windows 音频桥补充边界

deep-research-report 方案已经把旧公开 MVP 的“只支持麦克风”边界扩展为“麦克风 + 用户主动授权的 Windows 系统音频桥”。该能力必须满足以下约束：

- 独立页面入口：前端路由为 `/audio-bridge`，导航显示“会议监听”，负责配对码、连接状态、设备撤销、ASR 配置状态和故障诊断。
- 账号隔离：配对码、设备令牌和转写事件按当前登录账号隔离；登录、退出、切换账号后不得复用上一个账号的本地身份态。
- 明确授权：只有用户生成配对码并启动 Windows 音频桥后，才允许采集腾讯会议、飞书等客户端播放的系统音频。
- 原始音频不落盘：音频桥只向服务端 ASR 转发实时 PCM 分片；系统只保存 transcript、题词卡、报告和必要诊断状态。
- 失败不伪装：ASR 未配置时必须显示 `ASR_NOT_CONFIGURED` 或“语音服务未配置”，不能把本地规则或空转写显示成语音识别成功。

## 13. 交付闸口

后续进入实现前，必须以本方案为主，并同时满足：

- 所有 AI 输出都带来源或 fallback 状态
- 所有新接口都有 schema 校验
- 所有写路径都能触发检索资料增量刷新
- `npm run verify` 通过
